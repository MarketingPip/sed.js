What is sed?

sed (short for stream editor) is a Unix utility that transforms text via a compact scripting language. It processes data line-by-line without loading entire files into memory — making it indispensable for log parsing, batch text transformations, and automated editing pipelines.

Stream Processing

Processes text one line at a time with minimal memory footprint. Only a few lines reside in core at once — no temporary files needed.

Regex Engine

One of the earliest tools to support regular expressions. The s/// syntax influenced Perl, ECMAScript, and virtually every modern language.

Universal Presence

Preinstalled on every Linux, macOS, BSD, and Unix system. From embedded devices to cloud servers — sed is everywhere text needs transforming.

The "quick editor" is developed at Berkeley. It introduces regular expression pattern matching to interactive text editing — the conceptual ancestor of sed's regex engine.

Ken Thompson's ed

Ken Thompson creates the Unix line editor ed at Bell Labs. It introduces the s/// substitution syntax and / pattern delimiter that sed would inherit verbatim.

1973

grep is Born

Ken Thompson writes grep in a single night at Bell Labs. The g/re/p (global/regular expression/print) command from ed becomes a standalone tool — proving the power of pattern-based text processing.

one night of coding
the breakthrough
1973 — 1974
Origin

Lee McMahon Creates sed

At Bell Labs, Lee E. McMahon develops the stream editor. The original motivation was an analogue of grep for substitution — g/re/s. Foreseeing endless special-purpose variants, he built a general-purpose line-oriented stream editor instead.

Sed is a non-interactive context editor designed to be especially useful for files too large for comfortable interactive editing.

1979

Ships with Version 7 Unix

sed first appears in Version 7 Unix, becoming one of the foundational command-line utilities. It joins grep and awk as essential tools for shell scripting and data pipeline processing.

first public release
the power trio
1977

AWK Arrives

Aho, Weinberger, and Kernighan create AWK at Bell Labs. sed and AWK become "cousins" — sed for simple transformations, AWK for structured data processing. Together with grep, they form the Unix text processing trinity.

1988 — 1992

POSIX Standardization

The IEEE POSIX standard defines a portable specification for sed. This ensures consistent behavior across Unix variants — from Solaris to Linux to macOS — establishing the baseline that all implementations must support.

IEEE 1003.2
extensions era
1990s

GNU sed Extensions

The GNU Project develops its own sed implementation, adding powerful extensions: in-place editing (-i), extended regex support (-r/-E), the z command, and s///e (execute replacement). GNU sed becomes the default on Linux distributions.

-i -E s///e z
1990s

minised & Embedded

Eric S. Raymond reverse-engineers a minimal sed from 4.1BSD. minised becomes the GNU Project's interim implementation and remains vital for embedded systems and Minix due to its tiny memory footprint.

circular influence
1987

Perl & The Feedback Loop

Larry Wall releases Perl, explicitly inspired by sed and AWK. Perl's s/// and m// operators borrow directly from sed syntax. In turn, GNU sed later adopts Perl-compatible regex features.

2012

GNU sed 4.2.2

GNU sed 4.2.2 is released. The project continues under new maintainership, solidifying its position as the most feature-rich sed implementation and the default on virtually every Linux distribution.

Compatibility Guide

POSIX vs GNU

Understanding the two major sed dialects — and why Sed.js supports both.

POSIX sed

IEEE 1003.2 Standard

The portable baseline. Every Unix-like system must support these commands and behaviors. Use sed --posix on GNU systems to enforce strict compliance.

Core Commands
s, d, p, a, i, c, y, r, w, n, N, D, P, h, H, g, G, x, b, t, :, =, q
Basic Regular Expressions
Standard BRE syntax with groups and backreferences
Address Ranges
Line numbers, regex patterns, and /start/,/end/ ranges
No In-Place Editing
Must redirect output manually; no -i flag
$ sed --posix 's/foo/bar/g' input.txt

GNU sed

Linux Standard

The de facto standard on Linux. Adds powerful extensions for modern workflows while maintaining backward compatibility with POSIX scripts.

In-Place Editing -i
Edit files directly with optional backup suffix
Extended Regex -E / -r
ERE syntax: () groups, +, ?, | alternation
Execute Replacement s///e
Run the replacement as a shell command
New Commands
z, R/W, e, F
Case-Insensitive s///I
Match patterns without case sensitivity
$ sed -i -E 's/foo(bar)?/baz/g' *.txt

Sed.js Supports Both

Switch between POSIX strict mode and GNU extended mode with a single option. Every command is fuzz-tested against the reference implementations to ensure behavioral parity.

// POSIX mode
await sed('s/foo/bar/g', input, { mode: 'posix' });
// GNU mode with extensions
await sed('s/foo(bar)?/baz/g', input, { mode: 'gnu', extendedRegex: true });

Frequently Asked Questions

Who created sed and when?
sed was created by Lee E. McMahon at Bell Labs between 1973 and 1974. It was developed as a general-purpose stream editor, evolving from the concept of g/re/s (a grep analogue for substitution). The syntax was inherited from Ken Thompson's ed line editor (1971), with regular expression code written by Dennis Ritchie.
What is the difference between POSIX sed and GNU sed?
POSIX sed is the portable baseline defined by IEEE 1003.2 — every Unix-like system must support its core commands and basic regular expressions. GNU sed extends POSIX with in-place editing (-i), extended regex (-E), the z command, s///e (execute replacement), and more. Use sed --posix on GNU systems to enforce strict POSIX compliance.
What does sed stand for?
sed stands for stream editor. It is a non-interactive text editor that processes input line by line, applying a sequence of editing commands. Unlike interactive editors like ed or vi, sed reads from standard input (or files) and writes to standard output, making it ideal for automated text transformations and shell scripting pipelines.
Is Sed.js compatible with GNU sed?
Yes. Sed.js supports both POSIX and GNU sed modes. In POSIX mode, it follows the IEEE 1003.2 standard strictly. In GNU mode, it supports extensions like -i (in-place editing), -E (extended regex), s///e (execute replacement), and the z command. The library was validated with 100,000 fuzz-tested commands against both sed --posix and GNU sed reference implementations.
What is the relationship between sed, grep, and awk?
These three tools form the Unix text processing trinity, all born at Bell Labs. grep (1973) searches for patterns. sed (1974) transforms text via stream editing. AWK (1977) processes structured data with a full programming language. sed and AWK are often called "cousins" — sed for simple line-based transformations, AWK for columnar data and arithmetic. Together with grep, they cover virtually all command-line text processing needs.
